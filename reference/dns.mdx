Headscale supports Tailscale's DNS features including MagicDNS, global and split nameservers, custom search domains, and extra DNS records. All DNS settings are configured under the `dns` section of the configuration file.

<Note>
  For DNS configuration to take effect, Tailscale clients must have `--accept-dns=true` enabled. This is the default for Tailscale clients.
</Note>

## MagicDNS

MagicDNS automatically assigns DNS names to nodes on your tailnet. With MagicDNS enabled, nodes are reachable as `hostname.base_domain`.

```yaml
dns:
  magic_dns: true
  base_domain: example.com
```

The `base_domain` must be a fully qualified domain name (without a trailing dot) and must be different from your `server_url` domain.

### Override local DNS

By default, headscale overrides the local DNS configuration on each node and forces the use of headscale's DNS settings. To allow nodes to use their own local DNS configuration instead, set `override_local_dns` to `false`.

```yaml
dns:
  override_local_dns: true
```

## Nameservers

### Global nameservers

Global nameservers are used for all DNS queries that are not covered by split DNS rules.

```yaml
dns:
  nameservers:
    global:
      - 1.1.1.1
      - 1.0.0.1
      - 2606:4700:4700::1111
      - 2606:4700:4700::1001
```

### Split DNS

Split DNS routes queries for specific domains to designated nameservers, while all other queries continue to use the global nameservers.

```yaml
dns:
  nameservers:
    split:
      foo.bar.com:
        - 1.1.1.1
      corp.example.com:
        - 192.168.1.53
        - 192.168.1.54
```

## Search domains

Custom DNS search domains are appended to unqualified hostnames during DNS lookups. When MagicDNS is enabled, `base_domain` is always prepended as the first search domain automatically.

```yaml
dns:
  search_domains:
    - internal.example.com
```

## Extra DNS records

Headscale can serve additional DNS records through MagicDNS. This is useful for making internal services accessible by name without requiring separate DNS infrastructure.

A common use case is running multiple apps on the same host behind a reverse proxy, so services like Grafana are reachable as `grafana.myvpn.example.com` instead of `hostname:3000`.

<Warning>
  Only A and AAAA records are currently processed by Tailscale clients. Other record types (CNAME, TXT, etc.) are not supported.
</Warning>

Extra records can be configured in two ways:

<Tabs>
  <Tab title="Static entries (config file)">
    Use `dns.extra_records` for entries that do not change while headscale is running. Changes to these entries require a headscale restart.

    ```yaml
    dns:
      extra_records:
        - name: "grafana.myvpn.example.com"
          type: "A"
          value: "100.64.0.3"

        - name: "prometheus.myvpn.example.com"
          type: "A"
          value: "100.64.0.3"
    ```
  </Tab>
  <Tab title="Dynamic entries (JSON file)">
    Use `dns.extra_records_path` to point headscale at a JSON file. Headscale watches this file for changes and applies updates automatically without a restart.

    ```yaml
    dns:
      extra_records_path: /var/lib/headscale/extra-records.json
    ```

    The referenced JSON file should contain an array of records:

    ```json
    [
      {
        "name": "grafana.myvpn.example.com",
        "type": "A",
        "value": "100.64.0.3"
      },
      {
        "name": "prometheus.myvpn.example.com",
        "type": "A",
        "value": "100.64.0.3"
      }
    ]
    ```

    <Tip>
      If the JSON file is generated by a script, produce stable output by sorting keys consistently. Headscale uses a checksum to detect changes — unstable output triggers unnecessary reprocessing.
    </Tip>
  </Tab>
</Tabs>

### Verifying extra DNS records

After configuring extra records, verify they resolve correctly:

<Tabs>
  <Tab title="dig">
    ```bash
    dig +short grafana.myvpn.example.com
    ```

    Expected output:
    ```
    100.64.0.3
    ```
  </Tab>
  <Tab title="drill">
    ```bash
    drill -Q grafana.myvpn.example.com
    ```

    Expected output:
    ```
    100.64.0.3
    ```
  </Tab>
</Tabs>

### Example: reverse proxy with extra DNS records

The following NGINX configuration snippet shows how to route traffic for a service registered with an extra DNS record:

```nginx
server {
    listen 80;
    listen [::]:80;

    server_name grafana.myvpn.example.com;

    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $http_host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

## Disabling DNS management

To stop headscale from managing DNS on clients, set all fields under `dns` to empty values.
